# Требования к импортируемым данным

## 1. Поддерживаемые форматы и структура архивов

Приложение умеет обрабатывать одиночные файлы, каталоги с логами, а также **ZIP-архивы**.  
При загрузке архивов действуют строгие правила:

- В одном архиве могут находиться файлы **только одного формата**: либо только `.txt`, либо только `.csv`. Смешивать их нельзя.
    
- Внутри архива не должно быть посторонних файлов (исключение — автоматически игнорируемые системные папки вроде `__MACOSX` или скрытые файлы, начинающиеся с точки).
    
- При использовании формата `.csv` в архиве должен присутствовать хотя бы один файл со статистикой активности и хотя бы один с информацией о блокировках.
    

## 2. Правила именования файлов

Чтобы система могла отличить файлы активности сессий (`pg_stat_activity`) от файлов блокировок (`pg_locks`), имена файлов **обязательно** должны содержать специальные ключевые слова:

- **Для данных активности (Activity):** файл должен содержать в имени слово `stat` (без учета регистра). _Пример: `db_stat_data.csv`, `pg_stat_10.txt`._
    
- **Для данных блокировок (Locks):** файл должен содержать в имени слово `lock` (без учета регистра). _Пример: `db_locks_data.csv`, `pg_locks_10.txt`._
    

## Важное отличие для `.txt` форматов

Если вы используете текстовый формат (`.txt` из вывода консоли `psql`), **в имени файла обязательно должна содержаться дата и время** создания снимка (timestamp), так как внутри самих TXT-файлов время не указывается. Приложение поддерживает два паттерна:

1. `yyyy-MM-dd-HH-mm-ss` (Например: `pg_stat_2026-03-21-14-30-00.txt`)
    
2. `yyyyMMdd_HHmm` (Например: `pg_locks_20260321_1430.txt`)
    

_Для CSV-файлов дата в имени не обязательна, так как она извлекается из столбца `snap_ts` внутри самого файла._

## 3. Требования к содержимому (колонки и данные)

Ниже перечислен список столбцов, которые парсер ищет внутри файлов.

- Для **CSV**-файлов это должны быть заголовки в первой строке (разделенные запятой).
    
- Для **TXT**-файлов это стандартный табличный вывод утилиты `psql` (разделитель `|`), где в шапке указаны названия колонок.
    

## 3.1. Файлы активности (stat)

Эти файлы содержат информацию о текущих сессиях (выборка из `pg_stat_activity`). Ожидаемые колонки:

- `snap_ts` — время снимка _(обязательно только для CSV формата, например: `2026-03-21 14:30:00`)_.
    
- `pid` — ID процесса.
    
- `datid` — ID базы данных.
    
- `datname` — имя базы данных.
    
- `usename` — имя пользователя.
    
- `application_name` — имя приложения (отсюда программа может извлекать `Thread ID` и `Java Start Time`, если они передаются приложением).
    
- `client_addr` — IP-адрес клиента.
    
- `state` — текущее состояние сессии (active, idle, idle in transaction и т.д.).
    
- `query` — текст выполняемого SQL-запроса.
    
- `wait_event_type` — тип события ожидания (если применимо).
    
- `wait_event` — событие ожидания (если применимо).
    
- `xact_start` — время начала транзакции.
    

## 3.2. Файлы блокировок (locks)

Для файлов блокировок поддерживается два варианта структуры данных. Приложение автоматически определит тип по заголовкам:

## Вариант А: "Предвычисленные пары" (Precomputed pairs)

Это формат, в котором ваш скрипт сбора статистики уже сопоставил блокируемые и блокирующие процессы.

- `snap_ts` _(только для CSV)_
    
- `waiting_pid` — ID ожидающего (заблокированного) процесса.
    
- `locking_pid` — ID удерживающего (блокирующего) процесса.
    
- `waiting_mode` / `locking_mode` — режимы блокировок для обоих процессов.
    
- `waiting_query` / `locking_query` — SQL-запросы обоих процессов.
    
- `wait_event_type` / `wait_event` — события ожидания.
    

## Вариант Б: "Сырые данные" (Raw pg_locks)

Это прямая выгрузка из системного представления `pg_locks`. В этом случае приложение (IngestService) вычислит дерево блокировок и пары "кто кого блокирует" самостоятельно «на лету». Требуемые колонки:

- `snap_ts` _(только для CSV)_
    
- `pid` — ID процесса.
    
- `locktype` — тип объекта блокировки (relation, tuple, transactionid и т.д.).
    
- `mode` — режим запрашиваемой блокировки (ExclusiveLock, ShareLock и т.д.).
    
- `granted` — статус (`true` / `t` или `false` / `f`).
    
- Идентификаторы блокируемого объекта: `database`, `relation`, `page`, `tuple`, `virtualxid`, `transactionid`, `classid`, `objid`, `objsubid`.
    

---

**Резюме для корректного импорта:**

1. Если загружаете **CSV**, убедитесь, что в них есть колонка `snap_ts`, а в именах есть `stat` или `lock`.
    
2. Если загружаете **TXT** консольный вывод, убедитесь, что в именах файлов есть timestamp в поддерживаемом формате и слова `stat` / `lock`.
    
3. Следите за чистотой ZIP-архивов: не смешивайте форматы и не добавляйте посторонние файлы (иначе сработает защита и импорт будет отменен).
